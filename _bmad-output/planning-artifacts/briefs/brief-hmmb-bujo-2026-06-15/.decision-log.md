# Decision Log — BuJo Digital Brief

## Sessão: 2026-06-15

### Contexto inicial
- Documento base: `bujo-digital-planejamento.md` (sessão de planejamento pré-BMad de 25/05/2026)
- Intenção: usar o planejamento existente como ponto de partida, sem se prender a ele
- Modo: Create (novo brief formal)

### Decisões capturadas

**[D1] Problema central:** o gargalo é a replicação mecânica de estruturas (cabeçalhos, grids, gráficos), não as decisões de migração. Isso confirma o princípio de design do projeto.

**[D2] Custo quantificado:** ~40min/dia, 1h30/semana, 2h/mês, 1 dia/ano em setup mecânico.

**[D3] Dois sistemas paralelos:** caderno + Notion geram retrabalho de transcrição diário.

**[D4] Gatilho "por que agora":** nova função profissional que não permite o tempo perdido antes tolerável.

**[D5] Critério de sucesso ampliado:** fechar o caderno requer o ciclo BuJo completo (Daily + Weekly + Monthly + Future Log + migrações), não só o Dashboard Diário.

**[D6] Sistema de hábitos com pesos:** não é binário. Cada hábito tem um peso; o percentual de completude é ponderado. Diferencial real vs. qualquer app de hábitos do mercado.

**[D7] Saúde tem propósito médico:** os dados existem para comunicação com médicos em consultas. Módulo de relatórios médicos vai para backlog.

**[D8] Mobile é acesso real:** não só responsividade cosmética. Hugo frequentemente não tem o caderno em mãos — mobile é ponto de acesso genuíno.

**[D9] Multi-tenant desde o início:** Hugo planeja abrir o app para amigos. A premissa de "single user" do planejamento original está revogada.

**[D10] Ambientes dev e prod:** necessários dado o caráter multi-usuário.

**[D11] Tracking de alimentação por foto:** ideia registrada no backlog, não está no escopo do MVP.

**[D12] Módulo de relatórios médicos:** backlog explícito.

**[D13] Kindle View:** permanece no escopo, mas não é critério de abandono do caderno — é fase 3.

**[D14] Campos dinâmicos para Health, Medicines e Habits:** nenhum dos três é hardcoded. O usuário cria e desativa campos/entidades conforme a necessidade. Medicines é uma entidade própria (não um campo de Health) com lógica de agendamento e confirmação. Hábitos têm tipo de dado configurável e peso individual. O modelo de dados do planejamento original está revogado — padrão arquitetural (EAV, JSONB ou tabelas de definição+valor) a ser decidido na fase de arquitetura.
