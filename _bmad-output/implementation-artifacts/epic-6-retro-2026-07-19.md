## 2. O que foi bem

- **A previsão arquitetural das stories foi cirúrgica.** A 6.2 isolou `weight_at_time` na fórmula de completude *explicitamente para a 6.3*; quando a 6.3 chegou, a mudança para `peso_efetivo = weight × multiplier` foi uma troca de uma linha em `_completeness_pct`, não uma reescrita. A 6.2 também projetou `HabitDayEntry` para receber `day_type`/`multiplier_at_time`; a 6.3 só fez `AddField`. Isso é o pagamento de dividendos de fatias verticais bem desenhadas.
- **Zero CRITICAL/HIGH/MEDIUM no épico inteiro.** Quatro implementações sólidas; o `automate` (E2E) não encontrou nenhum bug de produção — um contraste com o Épico 5 (bugs reais em 2/3 stories). As stories da create-story vieram excepcionalmente detalhadas (decisões com defaults arq/AC-fiéis, refs precisas), o que tornou dev e review suaves.
- **Disciplina de contrato aditivo mantida.** O padrão `ENUM_NAME_OVERRIDES` (herdado da 6.1) evitou colisão de enums (`type` de habits × bujo; os dois `day_type` da 6.3 resolvidos com chaves nomeadas `weekend`/`holiday`; `field` do diff da 6.4 como CharField). Nenhuma regressão no contrato de `bujo`.
- **Testes reais mapeados às ACs + casos-âncora.** Idempotência de seed, imunidade de dias passados, não-sangramento, precedência holiday>weekend, completude com multiplicador=0, lacuna honesta ≠ 0%, cross-tenant 404, range bounds — todos cobertos. A 6.4 endereçou o follow-up de N+1 (query única + agrupamento em Python).
- **Regra de porta respeitada sob pressão** (6.3): `user_holidays` em `accounts` (não `habits`), lido por `core/calendar` via import tardio — a única casa que permite `core` ler sem violar `core ↛ habits`. `lint-imports` KEPT em todas as stories.

## 3. Onde a equipe tropeçou (sistemas e processos, sem culpar indivíduos)

- **Hazards do story-automator autônomo neste ambiente (o tropeço-mãe).** Três problemas distintos, todos de infra/tooling, não de código de produto:
  1. **Bug do `ensure-stop-hook` com espaço no path** (`Projetos Pessoais`): o `step-01-init` passava `--command "{scripts} stop-hook"` como string única; o `shlex.split` do helper fatiava o path no espaço e gravava um Stop hook corrompido (`Unknown command`), travando `monitor-session`. **Causa-raiz encontrada, reproduzida e corrigida** (args separados → `shlex.join`; commit `83d0266`). Era o que abortou a run original do Épico 6.
  2. **Remanescentes globais do pixel-agents:** hooks em `~/.claude/settings.json` em todos os eventos, resíduo de uma extensão já desinstalada. Removidos com autorização de Hugo (backup salvo).
  3. **Prompt fantasma** (`dev this story`, `commit this`, `yes run the report and commit`) aparecendo **não-submetido** na input box de sessões-filhas ociosas após conclusão. **Descartado pixel-agents como fonte** (hook nunca escreve stdout; sem servidor rodando). Benigno com kill imediato da sessão pós-conclusão — mas exigiu vigilância constante.
- **Flakiness/hang do Postgres remoto (Neon) — recorrente e agravado.** A suíte backend completa (não-escopada, serial) travou por >1h na 6.4 (conexão presa / lentidão do compute compartilhado, agravada por migrate concorrente). Mitigação `fileParallelism:false`/`workers:1` (da retro do Epic 5) ajudou mas não eliminou. **Padrão de contorno consolidado:** rodar em blocos foreground `--reuse-db` e, quando a full-suite trava, usar run **escopado** (habits+accounts+core) para a contagem honesta — foi o que deu os 222 passed da 6.4.
- **`dev-story` pausa para as "Decisões a confirmar" da story.** O build-cmd de dev não tem marcador `#YOLO`, então as sessões de dev pausavam para *perguntar* as decisões pendentes (o `unit` da 6.2; as 6 da 6.3; as 7 da 6.4). Numa sessão tmux destacada isso trava indefinidamente. **Contornado** injetando a decisão via `tmux send-keys` (endossar todos os defaults + YOLO) — funcionou nas três.
- **Assinatura de commit via 1Password travou** após ociosidade longa (assinatura SSH). Bloqueou o commit da 6.3 até Hugo desbloquear. Escalado corretamente (não commitei não-assinado unilateralmente, dado que os commits anteriores são assinados).
- **Gap de sync do sprint-status na 6.4:** a sessão dev marcou `Status: review` no story file mas não sincronizou `sprint-status.yaml` (ficou `ready-for-dev`); sincronizado na etapa de review. Ruído menor de bookkeeping.

## 4. Insights principais

- **O modo "dev controlado in-session" foi o modo confiável neste ambiente.** O orquestrador (esta sessão) dirige: spawna tmux **apenas** para create-story, dev-story e automate (tarefas longas que se beneficiam de isolamento), e faz o **gate de review de story, o code-review adversarial e o commit in-session**. Isso combina o isolamento do tmux para o dev longo com (a) review confiável e com contexto de épico completo, (b) escopo de commit preciso (resíduos de planejamento/UX do Hugo nunca varridos por engano), (c) neutralização imediata do prompt fantasma. É a lição operacional central do épico.
- **Injeção pró-ativa de decisão é o padrão para as pausas do dev-story.** Ler a seção "Decisões a confirmar" da story, endossar os defaults e enviar via `send-keys` *antes* de a sessão travar.
- **Run escopado é o fallback honesto de contagem** quando a full-suite do Neon trava — o que importa é que os testes da story (habits) passam; a contagem full-suite é registro, não gate.
- **Qualidade da story de create-story determina a suavidade do resto.** As 4 stories vieram com decisões documentadas + defaults + refs de arquivo precisas; isso é o que produziu 0 CRITICAL/HIGH e reviews rápidas.
