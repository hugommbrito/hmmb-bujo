## 2. O que foi bem

- **Dev Notes como contrato executável.** Cada story trouxe "forma normativa" (snippets de código prontos) para os padrões mais arriscados — o interceptor JWT single-flight (2.2), o `ProtectedLayout`/`LoginPageRoute` (2.3), o wrapper `Modal` (2.4). Resultado: zero retrabalho por ambiguidade de especificação nas 4 stories.
- **Disciplina de escopo quase perfeita.** Toda story trouxe uma tabela explícita "Pertence a esta Story / NÃO faça agora — Story responsável". Rastreando as 4 stories, nenhum item vazou de sua story planejada para outra — a Story 2.1 não tocou frontend, a 2.2 não tocou router/layout, a 2.3 não tocou a11y, e a 2.4 não introduziu nenhuma superfície nova.
- **A auditoria de acessibilidade centralizada na Story 2.4 funcionou como projetado.** A decisão de arquitetura de reservar uma story inteira para auditoria (em vez de checklist por componente) capturou um bug real e cross-cutting que as duas stories anteriores não viram: `<main>` aninhado entre `AppLayout` e `PlaceholderPage` — HTML inválido que quebra navegação por landmark em leitor de tela. Detectado só porque alguém revisou a casca inteira de uma vez.
- **Separação backend→frontend limpa.** Backend fechado 100% na Story 2.1 (79 testes); as 3 stories seguintes foram 100% frontend, sem nenhum vaivém entre camadas.
- **Aprendizado dentro do próprio épico.** O fix de ESLint boundary (regex de `no-restricted-imports`) aplicado na 2.2 nunca mais precisou de correção nas Stories 2.3/2.4 — o padrão pegou de primeira.

## 3. Onde a equipe tropeçou (sistemas e processos, sem culpar indivíduos)

- **Contagem de testes nas Completion Notes errada em 3 das 4 stories.** 2.1 relatou "69" quando o real era 79; 2.3 tinha uma contagem inconsistente corrigida para 99; 2.4 relatou "118" quando o real (verificado em execução serial) era 121. Padrão recorrente — não é acaso isolado. Isso não bloqueou nada (sempre pego em code review), mas é ruído evitável.
- **"0 erros de lint" declarado sem verificação, uma vez de forma crítica.** Na 2.4, o Debug Log afirmava `npx eslint src/` limpo, mas havia um import não utilizado (`Badge`) deixado de uma refatoração — só descoberto no code review. Como o guardrail de CI (`eslint`) já roda no pipeline, isso não teria vazado para produção, mas indica que a autoverificação do dev antes de marcar a task como concluída às vezes é otimista demais.
- **Uma AC da própria story quase falhou dentro da própria story.** A Story 2.4 existe para garantir "cor nunca é único indicador de estado" (AC1) — e a implementação inicial do item ativo da Sidebar (criada na 2.3, mas dentro do escopo de auditoria da 2.4) violava exatamente essa regra, usando só cor. Foi pego no code review final da própria 2.4, não achado pela "auditoria" documentada nos Dev Notes (que marcou a Sidebar como "✅ correto"). Sinal de que a auditoria de acessibilidade, mesmo bem estruturada, tem um ponto cego para o próprio critério que a motivou.
- **Flakiness de ambiente sob carga alta.** `npx vitest run` paralelo mostrou timeouts intermitentes em `router.test.tsx`/`SignupPage.test.tsx` quando o load average da máquina estava elevado (2.4). Não é regressão — confirmado com execução serial — mas é um ponto de atrito reproduzível que vale documentar como conhecido.

## 4. Insights principais

1. **Reservar uma story dedicada para acessibilidade no fim de um épico de UI captura bugs cross-cutting que revisões pontuais não capturam** — validado concretamente pelo bug do `<main>` aninhado.
2. **Guardrails automatizados (CI, lint, import-linter) são mais confiáveis que a autodeclaração do dev.** Toda vez que um dev relatou "0 erros"/"N testes" sem rodar o comando de novo antes de fechar a story, o número estava errado. O padrão que funcionou (Story 2.1: `import-linter` com `root_packages`) deve ser reaplicado sempre que um novo app Django for criado.
3. **Critérios de aceite sobre acessibilidade precisam de verificação automatizada (jest-axe), não só checklist textual** — a introdução de `jest-axe` na própria 2.4 é o que pegou a violação real de contraste/lista (`<ul>` com filhos inválidos no Sidebar). Onde havia teste automatizado, o achado foi real; onde havia só "auditoria visual" documentada em prosa (a tabela "CORRETO"), o ponto cego passou.
